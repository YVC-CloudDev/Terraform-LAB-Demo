Learning Objectives

By the end of this lab you will be able to:

  • Explain what Infrastructure as Code (IaC) is and why teams use it.
  • Author Terraform configuration in HCL and use the core workflow: init → plan → apply → destroy.
  • Provision an Amazon S3 bucket and configure it as a public static website.
  • Manage S3 public access blocks, bucket policies, and object uploads from Terraform.
  • Tear down all created resources cleanly to avoid AWS charges.
Outcome: A live HTTPS-reachable static site you provisioned entirely from code — just like the page you are reading right now.

What is Infrastructure as Code?

Infrastructure as Code is the practice of defining cloud resources — networks, virtual machines, databases, storage — in declarative configuration files that are checked into source control and applied by an automation tool. Instead of clicking through a cloud console, you describe the desired end state and let the tool reconcile reality with that state.

Why teams adopt IaC:
  • Repeatability: identical environments for dev, staging, and production.
  • Reviewability: infrastructure changes go through pull requests like any other code.
  • Auditability: Git history shows who changed what, when, and why.
  • Disaster recovery: rebuild an entire environment from code in minutes.
  • Cost control: destroy ephemeral environments when not in use.

Terraform by HashiCorp is the most widely adopted IaC tool. It uses a declarative language called HCL (HashiCorp Configuration Language) and supports thousands of providers — AWS, Azure, GCP, Kubernetes, GitHub, Datadog, and more — through a unified workflow.

Configuring AWS Credentials

Terraform needs AWS credentials to call the AWS APIs. Configure a named profile with the AWS CLI:

aws configure --profile lab

You will be prompted for four values:

  • AWS Access Key ID
  • AWS Secret Access Key
  • Default region name — for this lab use us-east-1
  • Default output format — json is fine
Security: never commit credentials to Git. Terraform will read them from your AWS CLI profile or from environment variables (AWS_ACCESS_KEY_ID, AWS_SECRET_ACCESS_KEY) at runtime.

Verify the profile works:

aws sts get-caller-identity --profile lab

Part 1 — Your First Terraform Resource

In this part you will create a single empty S3 bucket. The goal is to get familiar with the Terraform workflow before adding more pieces.

1

Create a working directory

mkdir aws-workshop
cd aws-workshop

Terraform treats every directory as an independent project with its own state. Keep each project isolated.

2

Create main.tf

Open VS Code in the current directory and create a file named main.tf:

code .
3

Declare the provider and a bucket

Paste the following into main.tf. Replace the bucket name with something globally unique — S3 bucket names share a single namespace across all AWS accounts.

terraform {
  required_version = ">= 1.5.0"

  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"
    }
  }
}

provider "aws" {
  region  = "us-east-1"
  profile = "lab"
}

resource "aws_s3_bucket" "site" {
  bucket = "yourname-tf-lab-2026"
}
Naming rules: lowercase only, 3–63 characters, no underscores, and globally unique across AWS. A good convention: <yourname>-tf-lab-<year>.
4

Run the Terraform workflow

terraform init

Downloads the AWS provider plugin and initializes the working directory. Run this once per project, and again whenever you change providers.

terraform init

terraform fmt

Rewrites your files into the canonical HCL style. Run before every commit.

terraform fmt

terraform validate

Checks syntax and basic semantic errors without contacting AWS.

terraform validate

terraform plan

Computes the diff between your configuration and the current real-world state. Always read the plan before applying.

terraform plan

terraform apply

Executes the plan. Terraform prints the diff again and asks you to type yes to confirm.

terraform apply
5

Verify in AWS

aws s3 ls --profile lab

Your bucket should appear in the list. You can also open the S3 console in your browser to see it.

Part 2 — Turn the Bucket Into a Website

An S3 bucket alone is just storage. To serve it as a public website we need four additional pieces:

  1. An index.html file to serve.
  2. A public access block configuration that allows public policies.
  3. A bucket policy that grants anonymous read access.
  4. A website configuration that designates the index document.
6

Create index.html

In the same directory, create index.html:

<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="UTF-8">
  <title>My Site</title>
</head>
<body>
  <h1>Hello from S3 + Terraform!</h1>
  <p>This page was deployed as code.</p>
</body>
</html>
7

Disable the public-access guardrail

By default, S3 blocks any policy that would expose objects publicly. Since we explicitly want a public website, we need to relax this guardrail. Append to main.tf: 

resource "aws_s3_bucket_public_access_block" "site" {
  bucket = aws_s3_bucket.site.id

  block_public_acls       = false
  block_public_policy     = false
  ignore_public_acls      = false
  restrict_public_buckets = false
}
Why the order matters: the public access block must exist before the bucket policy, otherwise AWS rejects the policy as a public-access violation. Terraform handles this automatically because the policy references the bucket — but be aware of the dependency.
8

Add a bucket policy for public read

resource "aws_s3_bucket_policy" "site" {
  bucket = aws_s3_bucket.site.id

  depends_on = [aws_s3_bucket_public_access_block.site]

  policy = jsonencode({
    Version = "2012-10-17"
    Statement = [
      {
        Sid       = "PublicReadGetObject"
        Effect    = "Allow"
        Principal = "*"
        Action    = "s3:GetObject"
        Resource  = "${aws_s3_bucket.site.arn}/*"
      }
    ]
  })
}

The jsonencode function lets us write the IAM policy as a native HCL object and converts it to JSON at apply time.

9

Upload the HTML and enable website hosting

resource "aws_s3_object" "index" {
  bucket       = aws_s3_bucket.site.id
  key          = "index.html"
  source       = "index.html"
  content_type = "text/html"
  etag         = filemd5("index.html")
}

resource "aws_s3_bucket_website_configuration" "site" {
  bucket = aws_s3_bucket.site.id

  index_document {
    suffix = "index.html"
  }
}

output "website_url" {
  description = "Public URL of the static website"
  value       = "http://${aws_s3_bucket_website_configuration.site.website_endpoint}"
}
About etag = filemd5(...): Terraform doesn't watch your local files. The etag argument forces it to re-upload the object whenever the file content changes, so every terraform apply publishes the latest version.
10

Deploy and visit your site

terraform apply

After confirming with yes, Terraform prints the output:

Outputs:

website_url = "http://yourname-tf-lab-2026.s3-website-us-east-1.amazonaws.com"

Open the URL in your browser — your site is live. Edit index.html, run terraform apply again, and refresh: changes ship in seconds.

Cleanup — Don't Skip This

S3 storage and requests are billed. When you finish the lab, destroy everything you created: 

terraform destroy

Terraform shows you the full deletion plan and asks for confirmation. Type yes and every resource defined in your configuration is removed. Verify with: 

aws s3 ls --profile lab

The bucket should no longer appear.

Troubleshooting

BucketAlreadyExists

Someone — possibly you, in another region — already owns that bucket name. S3 bucket names are globally unique. Pick a new one.

AccessDenied when applying the bucket policy

The public access block is still enforcing the restriction. Make sure aws_s3_bucket_public_access_block is in your configuration and that the policy depends_on it.

NoSuchBucket in the browser

Wait 10–30 seconds after terraform apply — S3 DNS propagation is fast but not instant. Also confirm the endpoint matches the bucket region.

403 Forbidden on the website URL

Three things to verify: (1) the bucket policy was applied successfully, (2) the public access block allows public policies, and (3) the object exists with the correct key (index.html, lowercase).

Provider plugin errors after editing main.tf

Run terraform init -upgrade to refresh providers.
